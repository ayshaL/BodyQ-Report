\section{Introduction}
% ─────────────────────────────────────────────────────────────────────────────

This chapter presents the development phase of the BodyQ mobile application,
from the technical decisions taken during the design phase through to the
complete working prototype. The objective of this phase was to transform the
requirements defined in the previous chapters into a fully functional,
AI-powered fitness application that runs on both iOS and Android devices.

The development of BodyQ was guided by three core principles:

\begin{itemize}
    \item \textbf{Personalization:} Every feature — from the workout plan to
    the daily meal suggestions — must adapt to each individual user's profile,
    goals, and physical condition.

    \item \textbf{Intelligence:} The application must go beyond static content
    by integrating real artificial intelligence that understands the user,
    learns from their activity, and responds meaningfully.

    \item \textbf{Usability:} A fitness app is only effective if users engage
    with it daily. Every screen, interaction, and flow was designed to be
    intuitive, fast, and visually appealing.
\end{itemize}

The prototype was developed as a cross-platform mobile application using
React Native with Expo, connected to a Supabase backend, and powered by two
AI providers: Groq (for conversational AI and plan generation) and Google
Gemini (as an alternative plan generation engine). The result is a complete,
deployable prototype covering user authentication, onboarding, AI workout and
meal planning, real-time tracking, analytics, community features, and an AI
coaching assistant named Yara.

\subsection{Application Architecture}

The application follows a layered architecture that cleanly separates
responsibilities across five layers, as illustrated in
Figure~\ref{fig:architecture}.

\begin{itemize}
    \item \textbf{Screens} — Full-page views that compose components and call
    hooks for their data logic.
    \item \textbf{Components} — Reusable UI pieces such as cards, progress
    bars, overlays, and modals.
    \item \textbf{Hooks} — Custom React hooks encapsulating business logic,
    one per feature domain (\texttt{useWorkout}, \texttt{useNutrition},
    \texttt{useInsights}, etc.).
    \item \textbf{Services} — All API calls and database operations
    (\texttt{workoutService}, \texttt{aiPlanService}, \texttt{profileService},
    etc.).
    \item \textbf{Library / Context} — Core utilities (Supabase client, Groq
    API wrapper, calculation functions, logger) and global state providers
    (Auth, Today, Fitness, Milestone).
\end{itemize}

This layered approach ensures that each layer only depends on the layer below
it, making the codebase maintainable, testable, and easy to extend.

\section{Development Support}

\subsection{Development Tools}

The following tools were used throughout the development process:

\begin{itemize}
    \item \textbf{Visual Studio Code} — Primary code editor with React Native
    and TypeScript extensions.
    \item \textbf{Expo Go} — Mobile application for testing on physical devices
    without building native binaries.
    \item \textbf{Expo CLI} — Running the development server and managing
    builds.
    \item \textbf{Supabase Studio} — Web dashboard for managing the database,
    writing SQL queries, and deploying serverless Edge Functions.
    \item \textbf{Git \& GitHub} — Version control using a feature branch
    workflow; each feature was developed on a separate branch and merged to
    \texttt{main} after review.
    \item \textbf{Postman} — Testing Groq and Gemini API endpoints during
    integration.
    \item \textbf{Android Emulator / iOS Simulator} — Testing on virtual
    devices without requiring a physical phone.
\end{itemize}

\subsection{Version Control Strategy}

Development used a \textbf{feature branch workflow}:
\begin{itemize}
    \item \texttt{main} branch: stable, deployable code at all times.
    \item Feature branches named by feature
    (e.g., \texttt{feat/community}, \texttt{feat/notifications} \texttt{scheduleweek-fix-generate}).
    \item Branches are merged into \texttt{main} after review and testing.
\end{itemize}

This approach prevented conflicts when multiple team members worked on
different features simultaneously, and ensured the main branch always remained
functional.

\subsubsection{Onboarding Flow}

New users complete a multi-step onboarding process that collects all
information needed to personalise the AI experience. The ten steps are:

\begin{enumerate}
	\item \textbf{Goal selection} --- Lose Fat, Gain Muscle, Maintain, Gain
	Weight, Build Habits, or Athletic Performance.
	\item \textbf{Body measurements} --- Height (cm), current weight (kg),
	target weight, and date of birth.
	\item \textbf{Experience level} --- Beginner, Intermediate, or Advanced,
	each with a description of the associated training approach.
	\item \textbf{Equipment} --- Full Gym, Dumbbells, Home (Bodyweight),
	Resistance Bands, or Kettlebells.
	\item \textbf{Schedule} --- Number of workout days per week (1--7), session
	duration (20--90 min), and preferred time of day.
	\item \textbf{Focus areas} --- Multi-select: Chest, Back, Legs, Arms,
	Shoulders, Core, Cardio.
	\item \textbf{Injuries and limitations} --- Multi-select: None, Knee,
	Back, Shoulder, Wrist, Ankle.
	\item \textbf{Diet preference} --- Anything, Vegetarian, Vegan, Keto,
	Gluten-Free, or Lactose-Free.
	\item \textbf{Lifestyle} --- Sleep quality (1--5) and stress level (1--5).
	\item \textbf{Completion screen} --- Animated confirmation before entering
	the main application.
\end{enumerate}

On completion, all data is saved to the \texttt{profiles} table, calorie
and macro targets are computed and saved to \texttt{calorie\_targets}, and
an interactive app tour launches to guide the user through the main tabs.

\subsubsection{Training Screen}

The training hub (Figure~\ref{fig:ui-train}) serves as the central hub
for all workout-related activity. From this screen, the user can:
\begin{itemize}
	\item Generate a personalised AI workout plan tailored to their
	profile, goals, and available equipment.
	\item Regenerate the plan with a preferred training split:
	Push/Pull/Legs, Upper/Lower, Full Body, Bro Split, Athletic,
	or Hypertrophy.
	\item Browse the exercise library filtered by muscle group or
	equipment category.
	\item Launch the \textbf{PostureAI} camera for real-time form
	analysis during exercises.
\end{itemize}

Additional workout screens support the full session lifecycle:
\begin{itemize}
	\item \textbf{WorkoutActive} --- guides the user through each
	exercise with a live rep counter and automated rest timer.
	\item \textbf{WorkoutSummary} --- displays post-session statistics
	including total reps, calories burned, duration, and average
	posture score.
	\item \textbf{PostureAI} --- overlays real-time skeletal landmarks
	on the camera feed and flags form errors as they occur.
	\item \textbf{FlappyBird Game} --- a mini-game available during rest
	periods to maintain user engagement between sets.
\end{itemize}

sleep duration, and calories burned — all synced from the mobile client
via BLE. The \textbf{Menu} screen provides navigation to three
sections: Workouts, Nutrition, and Insights. The \textbf{Workouts}
screen lists recent sessions started from the phone, and the
\textbf{Nutrition} screen displays macro targets (protein, carbs, fat)
and remaining daily calories, with meal logging redirected to the
mobile app.

\subsection{Feature Implementation}
\label{sec:features}

\subsubsection{Authentication and Session Management}

Authentication is handled by Supabase Auth. The \texttt{AuthContext}
provider wraps the entire application and manages the global user state.
On every app start, it calls \texttt{supabase.auth.getSession()} and
subscribes to \texttt{supabase.auth.onAuthStateChange()} to react
automatically to login and logout events.

When a user logs in, the following sequence executes:
\begin{enumerate}
    \item The user's training plan is fetched from Supabase and loaded into
    the local \texttt{scheduleStore}.
    \item If no Supabase plan exists, the local AsyncStorage is checked as
    a fallback.
    \item The user's profile is fetched to determine whether onboarding is
    complete (\texttt{profile.goal \&\& profile.onboarded}).
    \item A server-side RPC call (\texttt{record\_user\_visit}) updates the
    login streak in the \texttt{profiles} table.
    \item The local avatar is loaded from device storage.
\end{enumerate}

Navigation is determined by the following logic:
\begin{itemize}
    \item No active session $\rightarrow$ Sign In / Sign Up screens.
    \item Session exists, onboarding incomplete $\rightarrow$ Onboarding
    flow.
    \item Session exists, onboarding complete $\rightarrow$ Main application
    (five-tab navigation).
\end{itemize}

On logout, \texttt{scheduleStore.clear()} wipes the local plan from the
device to prevent data leakage between accounts.

\subsubsection{Onboarding and User Profiling}

On completion of the onboarding form, the \texttt{calculations.ts} library
computes the user's energy requirements using the Mifflin-St Jeor equations:

\begin{align}
    \text{BMR (male)} &= 10w + 6.25h - 5a + 5 \\
    \text{BMR (female)} &= 10w + 6.25h - 5a - 161
\end{align}

where $w$ is weight in kg, $h$ is height in cm, and $a$ is age in years.

The TDEE (Total Daily Energy Expenditure) is then computed as:
\begin{equation}
    \text{TDEE} = \text{BMR} \times \text{activity multiplier}
\end{equation}

The activity multiplier ranges from 1.2 (sedentary) to 1.725 (very active).
The daily calorie target is adjusted based on the user's goal:
\begin{itemize}
    \item Lose fat: TDEE $-$ 500 kcal/day
    \item Gain muscle: TDEE $+$ 250 kcal/day
    \item Gain weight: TDEE $+$ 500 kcal/day
    \item Maintain: TDEE
\end{itemize}

Macro targets are distributed as follows: protein at 2g per kg of body
weight (muscle gain) or 1.8g (fat loss), fat at 25\% of total calories, and
carbohydrates filling the remaining calorie budget.

\subsubsection{AI Workout Plan Generation}

The plan generation feature follows a five-step agentic workflow:

\begin{enumerate}
    \item \textbf{Profile retrieval} — Two Supabase queries run in parallel
    using \texttt{Promise.all()}: one fetching the user's profile and one
    fetching their calorie targets.

    \item \textbf{Prompt engineering} — The \texttt{buildPrompt()} function
    maps raw profile fields to fitness-specific instructions. For example,
    the experience level maps as follows:
    \begin{itemize}
        \item Beginner $\rightarrow$ ``simple compound movements, reps 12–15''
        \item Intermediate $\rightarrow$ ``compound + isolation, reps 8–12''
        \item Advanced $\rightarrow$ ``heavy compounds, reps 4–10, periodisation''
    \end{itemize}
    A random seed number is prepended to every prompt to ensure each
    generation produces a unique plan.

    \item \textbf{Groq API call} — The prompt is sent to
    \texttt{llama-3.3-70b-versatile} with \texttt{max\_tokens: 4096},
    \texttt{temperature: 0.8}, and \texttt{response\_format: json\_object}
    to enforce structured JSON output.

    \item \textbf{Defensive parsing} — The \texttt{parsePlanResponse()}
    function repairs common AI formatting errors: stripping markdown fences,
    removing trailing commas, fixing unquoted keys, removing control
    characters, and auto-closing any unclosed braces or brackets. A second
    parse attempt is made after repair before throwing an error.

    \item \textbf{Persistence} — The parsed plan is saved to
    \texttt{training\_plans} via an upsert operation with
    \texttt{onConflict: 'user\_id'}, ensuring each user has exactly one
    active plan. The plan is simultaneously written to the local
    \texttt{scheduleStore} so the Schedule screen updates without a reload.
\end{enumerate}

Regeneration cycles through six predefined training split styles using a
modulo index:
\[
\text{nextIndex} = (\text{currentIndex} + 1) \mod 6
\]
This guarantees structural variety (not just different exercise names) across
regenerations.

\subsubsection{Yara AI Chat Assistant}

Yara is the application's central conversational AI coach. Its system prompt
defines a personality (warm, direct, coach-like, no AI stiffness) and injects
the user's full profile — goal, weight, experience, equipment, injuries,
daily calories, and macros — at every call. Yara is instructed never to ask
for information already present in the profile.

The \texttt{useYaraChat} hook implements two output modes:

\begin{itemize}
    \item \textbf{Normal chat mode:} The user's message is sent to Groq
    \texttt{llama-3.1-8b-instant} with a 512-token limit for fast responses.
    The reply is displayed as a chat bubble and saved to
    \texttt{chat\_messages}.

    \item \textbf{Schedule generation mode:} Triggered by keyword detection
    (e.g., ``give me a weekly plan'', ``workout plan'', ``plan my week'').
    A strict JSON schema is appended to the user's message. The model
    responds with a full 7-day plan in JSON format, which is parsed and
    saved both to \texttt{scheduleStore} (for immediate UI update) and to
    the \texttt{training\_plans} table (for cross-device persistence).
\end{itemize}

Conversation history is persisted to Supabase and reloaded each time the
chat opens, allowing the model to maintain context across sessions.

\subsubsection{Schedule Store and Completion Tracking}

The \texttt{scheduleStore} is a custom lightweight pub/sub store that manages
the weekly plan state and exercise completion. Its state is structured as
follows:

\begin{itemize}
    \item \textbf{Schedule state:} The full AI plan JSON (days, exercises,
    meals, step and sleep targets).
    \item \textbf{Completion state:} A date-keyed map recording which
    exercises have been checked off and whether the day is marked done,
    e.g., \texttt{\{ "2026-05-13": \{ done: false, checked: [0, 2] \} \}}.
\end{itemize}

All writes are immediately persisted to AsyncStorage, making the schedule
available offline. Components subscribe to the store and re-render
automatically on state changes, without requiring a page reload.

Two adaptive behaviors are implemented in the Schedule screen:
\begin{itemize}
    \item \textbf{Exercise carryover:} If yesterday's workout was not
    completed, its uncompleted exercises are appended to today's list, tagged
    as ``carried over''.
    \item \textbf{Adaptive step targets:} If the user fell short of their
    step target the previous day, today's target is increased by the
    shortfall.
\end{itemize}

\subsubsection{Workout Tracking and Muscle Fatigue}

When a workout session ends, \texttt{saveWorkoutSession()} performs two
parallel database operations:

\begin{enumerate}
    \item \textbf{Session insertion} — A row is inserted into
    \texttt{workout\_sessions} with start/end timestamps, calories burned,
    and a notes summary (e.g., ``Squat · 12 reps · 78\% form'').

    \item \textbf{Muscle fatigue update} — Each exercise maps to a set of
    muscle groups (e.g., squat $\rightarrow$ Quads, Glutes, Hamstrings).
    For each targeted muscle, the stored fatigue percentage is increased by
    20 percentage points, capped at 100\%:
    \[
        \text{fatigue}_{\text{new}} = \min(100,\; \text{fatigue}_{\text{current}} + 20)
    \]
\end{enumerate}

Muscle fatigue data is displayed in the Insights screen and is also included
in the context sent to the Yara Insights AI, enabling recovery-aware
recommendations.

\subsubsection{Insights and AI Analysis}

The Insights screen aggregates data from five sources simultaneously using
\texttt{Promise.all()}: a Supabase RPC returning aggregated stats and a
42-day activity heatmap, direct food log queries for macro breakdowns, direct
workout session queries for training summaries, muscle fatigue records, and
cached AI insight history.

The \textbf{Health Score} formula assigns a daily score from 0 to 100:
\begin{align*}
    \text{Workout completed} &\rightarrow +35 \\
    \text{Steps} > 8{,}000  &\rightarrow +25 \quad (\text{steps} > 4{,}000 \rightarrow +12) \\
    \text{Food logged}       &\rightarrow +20 \\
    \text{Sleep} \geq 7\text{h} &\rightarrow +15 \quad (\text{sleep} \geq 6\text{h} \rightarrow +8) \\
    \text{Water} \geq 2\text{L} &\rightarrow +5 \\
    \hline
    \text{Maximum}           &= 100
\end{align*}

The \textbf{AI Insight Cards} are generated through a server-side Supabase
Edge Function to protect the Groq API key. A 6-hour cache is implemented
in the \texttt{ai\_insights} table: if four fresh rows exist, they are
returned immediately without calling the AI; otherwise, the Edge Function
is invoked, new cards are generated and stored, then returned to the screen.
A deduplication map prevents multiple concurrent requests from each
independently triggering a cache miss and calling the API redundantly.

\section{Conclusion}
% ─────────────────────────────────────────────────────────────────────────────

This chapter has presented the complete development and prototype of the
BodyQ mobile application. The implementation demonstrates that a
sophisticated, AI-powered fitness application can be built using modern
cross-platform tools while maintaining high code quality, a clean
architecture, and a smooth user experience.

The key achievements of this development phase are:

\begin{enumerate}
    \item \textbf{A complete agentic AI system} — Yara functions as a
    genuine AI coach: she understands the user's profile, generates
    structured weekly plans from natural language requests, and produces
    analytical insights derived from real behavioral data, all without
    requiring the user to understand the underlying technology.

    \item \textbf{Personalization at every level} — From the calorie ring
    on the home screen to the specific exercises in the weekly plan, every
    piece of content is derived from the individual user's profile. No two
    users see the same plan.

    \item \textbf{Robust offline capability} — The weekly schedule and
    exercise completion state persist locally on the device, meaning the
    core tracking features function without an internet connection.

    \item \textbf{Scalable, maintainable architecture} — The layered
    architecture (Screens $\rightarrow$ Hooks $\rightarrow$ Services
    $\rightarrow$ Lib) isolates each feature, making the codebase testable
    and replaceable without affecting the rest of the system.

    \item \textbf{Production-ready AI integration} — The application
    demonstrates practical AI engineering: structured prompt design,
    defensive JSON parsing, server-side API key protection via Edge
    Functions, intelligent result caching, and concurrent-request
    deduplication.
\end{enumerate}

The prototype is fully functional and covers all core user journeys defined
in the requirements. Future development will focus on expanding the exercise
library, refining the AI models with fitness-specific data, and introducing
social features such as group challenges and leaderboards.

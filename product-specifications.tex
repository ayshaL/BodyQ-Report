\chapter{Product Specifications}

\section{Introduction}
This chapter defines the full specification of the BodyQ project. It
covers the project management approach adopted by the team, the complete
product backlog, the sprint planning and execution, and the detailed
functional and non-functional requirements of the system. It concludes
with an analysis of the project's constraints, limitations, assumptions,
standards, and ethical considerations.

\section{Project Management}

\subsection{Methodology}
The BodyQ project was developed using an \textbf{Agile Scrum}
methodology. Scrum was chosen because the project required the integration
of several technically heterogeneous components---a mobile application,
an AI engine, a backend, and an administration dashboard---whose
interfaces and requirements were expected to evolve as implementation
progressed. Scrum's iterative sprint structure allowed the team to
deliver working increments at the end of each sprint, validate them
against expected behavior, and adjust the plan for the following sprint
based on what was learned.

For the Team of 5 students, sprint planning, review, and retrospective meetings were conducted at the beginning and end of each sprint. A
shared JIRA task board tracked user story status across To~Do,
In~Progress, and Done columns throughout the Backlog while tracking progress and contributions. Sprint duration was two weeks, with the
exception of Sprint~0, which was one week long.

\section{Sprint Backlog}
\subsection{Sprint 1}
Sprint~1 established the application's technical foundation. The team
delivered a full PostgreSQL schema spanning 18 tables covering user
profiles, workout sessions, food logs, daily activity, body metrics, and
AI insights. User authentication via email/password and Google
OAuth~2.0 was configured through Supabase Auth. A ten-step onboarding
questionnaire was developed to collect fitness goals, body metrics (height,
weight, and target weight), experience level, available equipment, preferred
weekly schedule, dietary restrictions, and lifestyle indicators (sleep
quality and stress level). Upon completion the questionnaire feeds a
Supabase Edge Function that invokes the \textbf{Groq Llama~3.1-8B} model
to generate a personalised multi-week training plan with calibrated daily
calorie and macronutrient targets derived from BMR and TDEE calculations.
On the administration side, the Next.js~14 dashboard was scaffolded with
Supabase service-role credentials and an initial user-management table view.

\subsection{Sprint 2}
Sprint~2 delivered the core tracking modules. The workout module was
completed with an exercise library, an active workout session screen that
records sets, reps, weight, and elapsed time per exercise, and a
post-workout summary. Nutrition logging was implemented with three
complementary input modes: text search across the food reference database,
barcode scanning via the device camera (\texttt{expo-camera}), and manual
macro entry. A water-intake tracker with a configurable daily goal ring and
a dedicated sleep log screen were also delivered. The home dashboard was
built to display real-time progress rings for calories, protein, water, and
step count, sourced from the \texttt{daily\_activity} PostgreSQL table
through parameterised Supabase RPC calls.

\subsection{Sprint 3}
Sprint~3 introduced the AI-powered features. The PostureAI screen was
built using an Expo WebView running MediaPipe Pose, with a custom
\texttt{PostureEngine} module computing joint angles and counting
repetitions in real time. The Yara RAG coaching assistant was
implemented as a Supabase Edge Function that retrieves seven days of
user activity, nutrition, and workout data before querying the Groq
API to generate contextualised coaching responses. AI food photo
recognition via the Gemini Vision API and a daily tip Edge Function
were also delivered.

\subsection{Sprint 4}
Sprint~4 extended the platform with advanced personalisation features.
A Pearson correlation engine was implemented to surface weekly
relationships between sleep, hydration, nutrition, and workout
performance on the Insights screen. Menstrual cycle tracking was
added with phase-aware adjustments to calorie targets, workout
intensity, and hydration goals. The administration dashboard was
extended with AI log review, aggregate platform analytics, and user
soft-deletion.

\subsection{Sprint 5}
Sprint~5 focused on stability, polish, and integration testing. Known
bugs were resolved, including the food scanner \texttt{.map} crash
caused by undefined API responses and a missing Gemini API key
configuration. UI animations were added, an \texttt{AppTour} onboarding
overlay was implemented, and end-to-end integration tests were run
across all modules to verify feature interactions before merging.

generated for both iOS and Android, and the administration dashboard

\paragraph{BOD-01 — Register Account.}
\textit{Actor}: unauthenticated user.
\textit{Precondition}: the user has not previously registered.
\textit{Main flow}: the user opens the registration screen, enters a valid
email address and password (or taps the Google sign-in button), and
confirms. Supabase Auth creates an account and inserts a corresponding row
in the \texttt{profiles} table with default values.
\textit{Postcondition}: the user is authenticated and redirected to the
onboarding flow.

\paragraph{BOD-02 — Complete Onboarding.}
\textit{Actor}: newly registered user.
\textit{Precondition}: the user has a valid account but no saved fitness
profile.
\textit{Main flow}: the user progresses through ten sequential screens
collecting goals (fat loss, muscle gain, maintain, or performance), body
metrics (height, weight, target weight), training experience, available
equipment, preferred days per week, session duration, time of day,
focus muscle groups, injury limitations, dietary preferences, and lifestyle
indicators. On submission the \texttt{onboarding-plan} Edge Function
invokes the Groq Llama~3.1-8B model to synthesise a personalised
$n$-day training plan; the plan and computed daily calorie and macro targets
are persisted in the user's profile.
\textit{Postcondition}: the user has a saved fitness profile and a
ready-to-use personalised training plan.

\paragraph{BOD-03 — Log Workout Session.}
\textit{Actor}: authenticated user.
\textit{Precondition}: the user has an active training plan.
\textit{Main flow}: the user navigates to the Training screen, selects the
current workout day, and starts the session. For each exercise the user
logs the number of sets, reps, and weight, and marks the set as complete.
When all exercises are finished the user ends the session; a workout summary
screen displays total duration and estimated calories burned.
\textit{Postcondition}: a \texttt{workout\_sessions} record and associated
\texttt{workout\_exercises} records are created; XP points are awarded and
recorded in \texttt{xp\_log}; muscle fatigue state is updated for the
trained groups.

\paragraph{BOD-04 — Log Nutrition.}
\textit{Actor}: authenticated user.
\textit{Precondition}: the user is authenticated.
\textit{Main flow}: the user opens the Nutrition screen and selects one of
three input methods---text search of the food reference database, barcode
scanning via the device camera (\texttt{expo-camera}), or AI photo
recognition powered by Google Gemini Vision. After identifying the food
item and confirming the portion size and meal type, the entry is saved.
\textit{Postcondition}: a \texttt{food\_logs} record is created; daily
calorie and macro totals on the home dashboard are updated.

\paragraph{BOD-05 — Log Daily Metrics.}
\textit{Actor}: authenticated user.
\textit{Precondition}: the user is authenticated.
\textit{Main flow}: the user taps a logging widget on the Home screen or
the dedicated Sleep Log screen, enters a value (water in millilitres, sleep
in hours, or body weight in kilograms), and confirms.
\textit{Postcondition}: the \texttt{daily\_activity} table (water and
sleep) or \texttt{body\_metrics} table (weight) is updated; the relevant
progress ring on the home dashboard reflects the new value in real time.

\paragraph{BOD-06 — Interact with AI Coach.}
\textit{Actor}: authenticated user.
\textit{Precondition}: the user is authenticated; Groq API access is
available.
\textit{Main flow}: the user opens the Yara coaching screen and types a
natural-language question. The \texttt{ai-assistant} Edge Function
classifies the query intent (activity, nutrition, workout, body metrics, or
history), fetches the relevant data subsets via Supabase RPCs, enriches the
context with semantically retrieved historical embeddings, and calls the
Groq Llama~3.3-70B model to generate a response. The response is returned
to the mobile client and persisted as an \texttt{ai\_insights} record.
\textit{Postcondition}: the user receives a data-grounded coaching reply;
the interaction is logged for subsequent coaching context.

\paragraph{BOD-07 — Use Voice Interface.}
\textit{Actor}: authenticated user.
\textit{Precondition}: the device microphone is available; internet
connectivity is active.
\textit{Main flow}: the user presses the Alexi voice button. A voice
activity detection loop records audio until silence is detected; the audio
is sent to the \texttt{ai-assistant} Edge Function, which transcribes it
via Groq Whisper (or OpenAI Whisper as primary). The transcript is
processed as a coaching query in voice mode. Any embedded
\texttt{COMMAND:\{...\}} blocks in the AI reply are parsed and executed
client-side (logging water, sleep, food, workouts, or triggering
navigation).
\textit{Postcondition}: the spoken command is executed without the user
touching the screen; a spoken audio response is played back via
\texttt{expo-speech}.

\paragraph{BOD-08 — Posture Detection.}
\textit{Actor}: authenticated user.
\textit{Precondition}: the device camera is available; the user is in an
active workout session.
\textit{Main flow}: the user activates the posture camera within the
WorkoutActive screen. A WebView loads the MediaPipe Pose estimation
pipeline, which processes camera frames and computes joint angles in real
time. Feedback cues (e.g.\ knee alignment, spinal curvature) and a live
repetition count are displayed as overlays. When the set ends, the session
receives an average posture score.
\textit{Postcondition}: the workout session record is updated with an
average posture score; the user has received actionable form corrections
during the exercise.

\paragraph{BOD-09 — View Insights.}
\textit{Actor}: authenticated user.
\textit{Precondition}: the user has at least some logged data.
\textit{Main flow}: the user navigates to the Insights screen and selects
a reporting period (Week, Month, or 3~Months). The screen calls the
\texttt{get\_insights\_data} Supabase RPC for chart and heatmap data,
queries \texttt{food\_logs} and \texttt{workout\_sessions} directly for
summary statistics, and fetches cached AI insight cards (or regenerates
them via the \texttt{yara-insights} Edge Function when the cache is stale).
\textit{Postcondition}: the user can view activity heatmaps, macro
breakdown charts, muscle recovery state, and four AI-generated insight
cards categorised by tag (Performance, Nutrition, Recovery, Correlation,
Optimisation, or Prediction).

\paragraph{BOD-10 — Administer System.}
\textit{Actor}: administrator.
\textit{Precondition}: the administrator has service-role credentials.
\textit{Main flow}: the administrator authenticates via the Next.js~14
web dashboard, browses the user registration table, and inspects AI
coaching interaction logs sourced from \texttt{ai\_insights}.
\textit{Postcondition}: the administrator has visibility into platform
usage, user registrations, and AI behaviour without direct database access.

\section{Project Considerations}

\subsection{Project Constraints}

The BodyQ project operated under three primary constraints.

\begin{enumerate}
	\item \textbf{Time.} The project was delivered across a fixed
	nine-week academic schedule divided into five sprints. No schedule
	extension was available.
	
	\item \textbf{Budget.} All infrastructure services were operated on
	free tiers. Supabase free tier provides 500~MB of managed PostgreSQL
	storage and 500{,}000 Edge Function invocations per month. Groq API
	access was limited to the free research tier rate limits. No financial
	resources were available for paid cloud services or proprietary data
	providers.
	
	\item \textbf{Team capacity.} The project was delivered by five
	students who were simultaneously managing other academic commitments,
	constraining the total development hours available per sprint and
	requiring disciplined prioritisation of backlog items.
\end{enumerate}

\subsection{Project Limitations}

Several functional limitations exist in the current prototype.

\begin{itemize}
	\item \textbf{Posture detection performance.} MediaPipe Pose is
	delivered inside a React Native WebView rather than a native module,
	because no production-grade React Native binding for MediaPipe was
	available at development time. This introduces a bridge communication
	overhead and constrains the usable camera resolution.
	
	\item \textbf{Voice interface connectivity.} The speech-to-text
	pipeline relies on Groq Whisper or OpenAI Whisper cloud services.
	The voice interface is non-functional without active internet
	connectivity.
	
	\item \textbf{Food database accuracy.} Nutritional information is
	sourced from third-party databases and user-entered data, which may
	contain inaccuracies. The system cannot independently verify the
	correctness of nutritional values.
	
	\item \textbf{Wearable integration.} Sprint~5 explored wearable
	device data ingestion but a production-quality integration was not
	completed within the project timeline.
	
	\item \textbf{Correlation engine scope.} The behavioral correlation
	engine surfaces statistical associations between health metrics. It
	does not account for confounding variables or apply clinical-grade
	statistical methods.
\end{itemize}

\subsection{Project Delimitations}

The following are deliberate scope boundaries established at the start of
the project.

\begin{itemize}
	\item The system is scoped to a mobile application (iOS and Android)
	and a web administration dashboard. No desktop client is planned.
	
	\item The application does not integrate with electronic medical record
	(EMR) systems and does not provide clinical diagnosis or treatment
	recommendations.
	
	\item Real-time social or multiplayer features (workout-partner
	streaming, group challenges) are outside the scope of the current
	prototype.
	
	\item The posture detection module supports a representative subset of
	exercises in its MVP form and does not cover the full exercise library.
\end{itemize}

\subsection{Project Assumptions}

The following assumptions were made during the design and development of
BodyQ.

\begin{itemize}
	\item Users possess a smartphone running iOS~16+ or Android~11+ with a
	functional camera and microphone.
	
	\item Users have reliable internet connectivity when using AI-powered
	features such as coaching queries, voice interface, and food photo
	recognition.
	
	\item Users understand that AI coaching responses are informational
	guidance and not a substitute for professional medical or nutritional
	advice.
	
	\item Nutritional data returned by third-party sources is sufficiently
	accurate for consumer-level calorie and macronutrient tracking.
	
	\item Users are willing to log health data manually when automatic
	detection is unavailable.
\end{itemize}

\subsection{Project Standards}

The following standards and conventions were applied throughout the project.

\begin{itemize}
	\item \textbf{Project management.} Agile Scrum with two-week sprints,
	sprint planning, sprint review, and retrospective ceremonies tracked on
	a shared JIRA board.
	
	\item \textbf{Code style.} ESLint and Prettier enforced uniform
	formatting across all TypeScript and JavaScript source files.
	
	\item \textbf{API design.} REST conventions applied to all Supabase
	Edge Function endpoints; CORS headers configured for cross-origin
	mobile and web access.
	
	\item \textbf{Database security.} Supabase Row Level Security policies
	applied to all user-data tables, aligned with GDPR data-minimisation
	and access-control principles.
	
	\item \textbf{Citation.} IEEE numeric reference style, ordered by
	first appearance in the text.
\end{itemize}

\subsection{Business, Social and Ethical Considerations}

\textbf{Privacy and data protection.} BodyQ collects sensitive health data
including body weight, sleep duration, nutritional intake, and (optionally)
menstrual cycle information. All personal data is stored in a Supabase
PostgreSQL database governed by Row Level Security policies that restrict
each authenticated user to accessing exclusively their own records. No user
health data is transmitted to or shared with third parties beyond the AI
inference providers strictly required for core functionality (Groq, OpenAI,
Google Gemini).

\textbf{AI transparency.} All AI-generated content---coaching messages,
personalised workout plans, meal suggestions, and insight cards---is
visually distinguished within the application interface. Users are not
presented with AI-generated output as if it were advice from a human
expert.

\textbf{Health responsibility.} The application explicitly positions its
coaching as informational guidance. Onboarding screens and AI coaching
responses do not claim to provide medical diagnosis, clinical treatment,
or prescribed nutritional therapy. Users are encouraged to consult
qualified healthcare professionals for any medical decisions.

\textbf{Inclusivity.} The onboarding questionnaire accommodates a wide
range of user profiles: multiple fitness experience levels, diverse dietary
preferences (vegetarian, vegan, halal, and other restrictions), varying
equipment availability (home, commercial gym, bodyweight-only), and
optional menstrual cycle tracking. This ensures the system does not
structurally exclude underrepresented user groups.

\textbf{Data minimisation.} Only the health metrics directly required by
the coaching and analytics features are collected. No location data, social
network information, or financial data is requested or stored.
